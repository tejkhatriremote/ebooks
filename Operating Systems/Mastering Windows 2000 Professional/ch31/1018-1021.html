Performance Options  Clicking the Performance Options button opens the Performance Options dialog. As the name implies, the two settings on this page affect system performance—in most cases, changing these will have an adverse impact on performance, so use with care. The first control is a pair of radio buttons that adjust how much extra processing time is given to the foreground, or focused application, as opposed to any other applications running in the background. The default setting of Applications gives the foreground application maximum priority, which is good if the primary use of your system is as a workstation. If your system will be used primarily as a server, however, then it makes sense to click the Background Services radio button so that background applications get the same priority as any foreground application. The following graphic shows the radio buttons for Application Response. Press the Change button shown in the graphic to reach the virtual memory settings.

The section labeled Virtual Memory shows you how much disk space is being used for all paging files. Unlike Windows 9x and Windows 3.1, Windows 2000 lets you have multiple paging files—one for each drive, in fact. This scheme enables Windows 2000 to increase disk performance by using a paging file located on one drive while accessing data located on another drive. As you can see in Figure 31.11, you can set the paging file’s current and maximum sizes for each drive.


Figure 31.11  To improve performance, Windows 2000 can have separate paging files on each physical drive.

Another setting you can adjust here is the maximum Registry size. Note that this does not allocate the specified amount of space, and therefore it does not guarantee that the space will actually be available should the Registry ever need it. Instead, it simply sets the upper limit on the amount of disk space the system Registry files will be permitted to use. This is another good reason (besides affecting general system performance) for why you should try to avoid having your drives fill up completely.

Environment Variables  If you’ve used NT, Windows, DOS, or for that matter, even Unix, you are probably familiar with the idea of environment settings. An environment setting, also known as an environment variable or system variable, comprises a setting name and its value. Certain applications, or the operating system itself, occasionally need to look at these settings. One of the most common of these variables is PATH, which holds a concatenated list of all directories the system should search when looking for an application or dynamically loaded library file. To change an existing setting, select it in either the System or User Variables column, and then click the Edit button. In the Edit System Variable (or Edit User Variable) dialog box, go down to the Variable Value field, enter the new value, and click OK. Similarly, to add a new setting click the New button, type the name of the new variable, enter the value in the Value field, and click OK. Deleting a setting is even easier: Simply highlight the setting and click the Delete button.

Eventually, we believe environment variables are going to be phased out completely in favor of using the System Registry. But for now, these variables provide a useful way for login scripts, DOS programs, and certain Windows applications to access information about the system. Startup and Recovery  The upper portion of this dialog box is where you select the operating system that will boot next time. As you can see in Figure 31.13, you can specify how long at bootup the operating system menu should wait for a user to change this setting before proceeding to boot the operating system selected in this dialog box. This is mainly useful when you have several versions of NT on your system, or both Windows 9x and Windows 2000. If you need to select from among other (non-Microsoft) operating systems, you will need a more extensive third-party boot manager, such as the System Commander utility. Notice that there is a new check box in this section that tells Windows 2000 whether it should display the boot menu at all. If you uncheck the Display List of Operating Systems setting, the boot menu won’t be displayed. This has the same effect as setting the timeout value in the boot.ini file to zero.


Figure 31.13  The Startup and Recovery dialog box lets you pick which operating system will boot when the system is rebooted.

The lower portion of this dialog box has to do with how the system behaves when encountering a serious system error. Although these errors are quite rare, if a system process or thread is forced to terminate unexpectedly, Windows 2000 may become unstable; when this happens, usually the safest action is to restart the system (the last choice, Automatically Reboot). In addition to choosing whether to have the system automatically restart, you can also have any combination of a logged event, administrative alert, or memory dump be performed automatically when such an error occurs.
