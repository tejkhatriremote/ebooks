Using the Novell Administration Utilities

To help you set up these myriad directories—users directory, trustee directory, and file attributes—NetWare comes with a complete set of network utilities to help you set up access rights. Of these, the main utility for administrative purposes is known as the system configuration utility (SYSCON for short) for NetWare 2.x and 3.x and NETADMIN for NetWare 4.x (NWADMIN is the Windows version of this utility). NetWare 5.x brings a new administration tool, ConsoleOne. With SYSCON and NETADMIN, you can set up accounting and metering features and set time restrictions for user logons. For example, let’s say that Jane has rights to use the file server only on Monday through Friday, from 8:00 AM to 3:00 PM. If she tries to log on at 4:00 PM Friday she will get an “access denied” or “invalid logon” error message. ConsoleOne is a one-stop tool for administering both a NetWare 5 server, and NDS. Accessed from a NetWare client, ConsoleOne gives you full access to the administration duties of your NetWare network with one tool. ConsoleOne is shown in Figure 25.1.


Figure 25.1  The ConsoleOne tool enables you to manage your NetWare 5 network from Windows 2000.

With ConsoleOne you can create users, groups, and Organizational Units (OUs). You can set trustee rights for the containers in your NDS tree, and set directory rights. Essentially, the functions of SYSCON and NETADMIN have been combined into a single tool that can be run either from the new GUI of NetWare 5 or from any Windows-based client. That’s the good news.

The bad news for people who want to use Windows 2000 Professional to manage their NetWare 5 network is that the tool does not have full functionality with Microsoft’s Client Services for NetWare. As with NT 4 and earlier, you must map a drive letter to access the Sys volume before you can run any NetWare administration tool. UNC paths will not work at all with these tools. If you are serious about managing your NetWare environment from within Windows 2000, you should strongly consider downloading Novell’s Client32 when it is available for Windows 2000.

Talking to NetWare 3.x Servers vs. Talking to an NDS Tree

Face it; you have to be aware of some facts of life regarding Novell and its three primary network operating systems—NetWare 3.x, 4.x, and 5.x. The heart of the difference lies in the way that NetWare versions view and handle security and validation of users. NetWare 4.x and 5.x both use NetWare Directory Services (NDS), which is an X.500 directory structure, very much like Windows 2000’s Active Directory.

A NetWare 3.x server holds its security information in a security database called the bindery. The bindery method of security views each server as an individual entity and the security that you assign to various directories and print queues is kept independently on each server. On the other hand, the NDS method of security that NetWare 4.x and 5.x use no longer treats each server as an independent unit, but instead treats a conglomeration of servers as a single unit called a tree. As a matter of fact, we really don’t care about the individual server in NetWare 4.x and 5.x. Instead we look to the different resources or objects that are being offered on each server, and the NDS tree keeps track of these objects and controls access to them.

To illustrate the difference, assume that you had to build 100 user accounts on 5 NetWare servers and that each of the 100 users needed access to the 5 NetWare servers. If you were building these accounts in a NetWare 3.x environment, you would have to build 500 user accounts—each of the 100 users on each of the 5 servers. If you were building these accounts in a NetWare 4.x or 5.x environment, you would have to build only 100 user accounts because all 5 servers would be part of the same NDS tree; you have to log on to an NDS tree only once.

The reason that this difference is important to you at the Windows 2000 Professional level is that the levels of access and control that you have on a NetWare 3.x server are different than on a NetWare 4.x or 5.x server. Therefore, a Windows 2000 machine interacts with NetWare 3.x and 4.x or 5.x in different fashions.

Accessing a NetWare Server

After you install the Client Service for NetWare (discussed later, in the section “Configuring Windows 2000 to Run in Parallel with NetWare”), you can access the resources of a NetWare 3.x server as well as a NetWare 4.x or 5.x tree, as long as you have a valid login ID on the NetWare servers. The login scripts that reside on the Novell server will run even on your Windows 2000 workstation, mapping your drives and capturing your printer ports.

Administering a NetWare Server from a Windows 2000 Computer

After the Client Service for NetWare is installed, you can still go to the command line prompt and run SYSCON (the system console) on a NetWare 3.x server in order to build users, manage groups, and assign trustee rights to the various resources this NetWare server has to offer. Life proceeds normally as long as all you want to do is administer a 3.x server.

However, if you turn your attention to a NetWare 4.x server, life is much different. You cannot run the NETADMIN or NWADMIN utilities to manage a NetWare 4.x NDS tree. At this time the Client Service for NetWare does not support logging on to the NDS as administrator to allow direct administrative access to the NDS. Look for this to change in subsequent patches for this service, especially in the Service packs that Microsoft will release for this product.

WARNING:  Many administrators may have gotten around the NDS administration limitation in NT Workstation 4 with 32-bit Novell Administrator software provided by Novell. This utility does not work on Windows 2000. In most cases, it will bring your system to a screeching halt, generally making it difficult (if not impossible) to reboot Windows 2000. Look on Novell’s Web site (www.novell.com) for the Windows 2000 version of the utility.

NetWare 5.x uses the ConsoleOne tool to manage the network resources from a client computer. Currently, this tool won’t work correctly with Windows 2000. To really use this tool’s functions, you will need to install the Client32 software from Novell. The key issue appears to be some proprietary functions supplied in some dynamic link libraries (DLLs) that are installed with the Client32 software.
