Accessing NetWare Servers from Windows 2000 Clients

If your business is like thousands of others around the world, there’s a good chance you have a Novell NetWare network already in place. And if you’ve just made the move to using Windows 2000 Professional, you’ll be pleased to know that it’s not difficult to connect your Windows 2000 computer to the Novell NetWare server. In other words, you’ve come to the right chapter.

Microsoft provides a special solution for this situation, which it calls the Client Service for NetWare. The benefits of the Client Service for NetWare include the following:

Novell script processing  The Client Service for NetWare processes the system and individual logon scripts created by the network administrator of the Novell network. Please note that the Client Service for NetWare still has limited support for NetWare Directory Services (NDS) in this iteration of Windows 2000; therefore, the script processing takes place when logging onto a NetWare 4.x or 5.x server.
Long filename support  If you enable OS/2 naming support on your Novell server, you can save files with names up to 254 characters in length. With the OS/2 naming support enabled, your NetWare server appears as a High Performance File System (HPFS) volume to the Windows 2000 client.
Single peripheral systems  Because you are using the 32-bit drivers for connectivity, you can use the dial-up services that are part of the Microsoft enterprise network. Historically, Microsoft products have referred to the dial-up services as Remote Access Service (RAS), but in Windows 2000 Professional, this function is known as Dial-Up Networking. If you use the Client Service for NetWare, dial-up networking will enable you to attach directly to the Novell servers and process NetWare logon scripts remotely.

Before we look at the procedure for installing the Client Service for NetWare, let’s look at the different versions of NetWare to which you may have to connect—NetWare 3.x, 4.x, or 5.x. We will also look at the printing resources that are available to you after you install the Client Service for NetWare. We conclude this discussion with some notes about utilizing logon scripts and mapping drives to the NetWare servers.

Defining Novell’s Point of View: Common Terms

A lot of the concepts that you are familiar with in Windows 2000 are present in Novell NetWare. Let’s take a moment to clarify the Novell terms.

Novell Security

NetWare security distinguishes four security issues:

•  The login procedure
•  Trustee rights
•  Directory rights
•  File attributes

Each of these levels provides a different and more complete level of security.

The Login Procedure

Every time a user logs on to the network, she must provide her identification name and a password. This password is NetWare’s guide to the access rights that the user has on the network. If the password has read-only rights attached to it, then that user can’t modify files on the server. Conversely, if that password has full rights to a file server and all file servers are treated as one on the network (as in NetWare 4.x or 5.x), then that user has full rights to all servers.

Trustee Rights

Each user or group of users on a Novell network is considered a trustee. Each user has a list of possible rights or permissions linked to his or her password (as we saw above), including but not limited to:

•  Reading from open files
•  Writing to open files
•  Opening existing files
•  Creating new files
•  Deleting existing files
•  Setting trustee rights for others at the directory/subdirectory level
•  Modifying file attributes

Although these rights can apply evenly to entire disks, they don’t necessarily have to. A network administrator can vary a user’s rights according to a particular directory of a drive. In other words, you could have full access to most of a drive, but read-only access to a particular directory on that drive. This concept leads us to our next topic—directory rights.

Directory Rights

Directory rights are similar to the directory permissions in Windows 2000. Although NetWare, by default, gives child directories (subdirectories) the same rights as their parent directories, each trustee can set the above rights for each directory and subdirectory on the network. Subdirectories do inherit the rights of their parent directories, just as they do in Windows 2000.

File Attributes

As a final level of security, the network administrator can mark individual files with certain attributes that limit how and when users can use them, similar to setting file-level permissions in Windows 2000. (NetWare just has fewer of them.) These attributes are:

•  Shareable vs. non-shareable. If a file is shareable, then if one person is using that file, others can access it as well. If a file is non-shareable, then no more than one person at a time can use (even read) that file.
•  Read/write vs. read-only.

You’re probably pretty familiar with read/write and read-only file attributes. If a file is read-only, then any user with those rights can read it and make changes but can’t save those changes to the original document. He could save those changes if he saved the amended file under another name, however. This kind of restriction is handy for a group project in which one person is in charge and other people are contributing. If the helpers can’t modify the original, the person in charge has to incorporate only the changes that she approves. This process saves the helpers from frantically looking for the original file’s backup after modifying the file in a way in which the project leader rejected. (And unless the helpers made a backup copy, the old file would no longer exist.)

Any user with read/write capability (such as that project manager) could update a file and then save it under the same name as before, incorporating the new information. Again, a backup file must be made or the old file will no longer exist.
