The typical life of a KB article begins when an engineer hangs up the phone after fixing a new problem. “Wow,” he exclaims, “that was the weirdest thing I’ve seen yet.” The engineer mentions the issue to his neighbor, who starts taking notes because she’s never seen it either. Then the local KB Editor hears about it and starts pestering the engineer to write it up in an article. Now, not every engineer can or will write articles, but there is always someone looking to expand their writing career who will tackle the article if the engineer can explain it well enough. So one way or another the article gets written. Then it is edited for style and reviewed for technical accuracy. If it passes all the reviews, it is added to the Knowledge Base.

The support engineers at Microsoft use these articles to resolve issues for customers, and once in a while the engineers find an article that isn’t quite accurate. They can then submit a comment to the editors describing the problem with the article so it can be fixed. On occasion the comments are helpful; sometimes they’re just very emotional. The point is, these articles are constantly being reviewed and updated for accuracy.

Accessing the Knowledge Base

The Knowledge Base can be accessed in a few ways. The first is to call Microsoft Technical Support and let them do the browsing through the KB. This might be a better answer than you realize because they have access to articles that aren’t released to the public. This isn’t some dark Microsoft conspiracy; it’s mostly out of consideration for other companies’ reputations (and their own). If a problem is serious but hasn’t been fully documented by all parties involved, Microsoft isn’t going to expose itself to possible litigation over a KB article. The information will be marked as MSONLY or PARTNER so that only support engineers can see the information. After all parties have confirmed the issues, the article can be released to the public.

The easiest way for us mere mortals to use the Knowledge Base is to access it through the Support Online pages at Microsoft’s Web site. This location is updated approximately twice a week to contain the latest information. If there is a hot issue that needs to be dealt with immediately (such as a bug in Internet Explorer’s security), the fix is released overnight and an article is sent into the KB right away.

The third way is to use TechNet, which is a subscription-based product from Microsoft for Information Systems and support professionals. TechNet is sent out once a month on several CD-ROMs. The first CD-ROM contains the entire public Knowledge Base, additional white papers, Resource Kits, and training materials. The remaining CD-ROMs contain the entire Microsoft driver library; all the current Service Packs, patches, and device drivers that Microsoft provides for their operating systems; and other products.

Understanding the Art of Query

Finding information in the Knowledge Base can be difficult. Face it, this thing is huge. Microsoft support engineers spend time in training after they’re hired just learning to use the KB effectively. Don’t feel bad if you have trouble at first finding the information you want. We’ve listed some helpful hints here that should make the task easier.

Start specific, and move outward. Try querying on a specific error message (if you have one). If that doesn’t work, try a few words from the error and the names of the files or devices involved in the problem. For example, if you wanted to find an article about a blue-screen error and your Adaptec 2940 controller, you might submit this query:

Winnt and adaptec and 2940 and STOP and 0x1E

And if there were any issues related to a blue screen with the STOP code 0x1E dealing with an Adaptec 2940 controller, the list of articles would be displayed (there aren’t any, in case you’re wondering). Notice the and between each of the query words. When querying the KB using this structure, the results will be articles that contain all the words. You are literally asking for all articles containing the words Winnt, Adaptec, 2940, STOP, and 0x1E. Any article that has only some of the words will not be returned. The default action when you type a string of query words is to find any article that contains at least one of those words.

There is a way to increase the likelihood of success in querying the KB. If you include certain keywords in your search, you will find specific articles more easily. In Table 32.2 we have included some keywords that can help improve your KB query skills.

Table 32.2 Knowledge Base Keywords for Windows NT
KB Keyword Description
Kberrmsg Returns articles dealing with error messages. Can be used with all products.
Kbsetup Articles dealing with setup issues. Can be used with all products.
Kbtool Concerns articles about commands or utilities. Can be used in all products.
Ntboot Articles dealing with the NT boot process. NT specific.
Ntdirlist Directory listing of distribution disks. NT specific.
Ntdomain Deals with NT domain administration. NT specific.
Ntras Articles concerning Remote Access Service. NT specific.
Ntsetup Returns articles pertaining to NT Setup issues. NT specific.
Ntsp Lists articles with information about NT Service Packs. NT specific.
NtSrv Articles that apply only to NT Server. NT specific.
NtSrvWkst Articles that apply to both NT Server and NT Workstation. NT specific.
Ntstart Boot process and start-up of NT. NT specific.
Ntstop Articles concerning STOP screen messages or blue screens. NT specific.
NtWkst Articles that apply to NT Workstation only. NT specific.
