Rock-Solid Multitasking

I alluded to Windows 2000’s multitasking abilities earlier in the section on stability, but it’s worth expanding upon. Windows 2000 is built upon a kernel (the basic engine of any operating system) that is designed to ensure that multiple tasks all get serviced. One of the benefits of multitasking is that you’ll have fewer system lockups.

Have you ever asked your PC to do something, only to have it freeze up? Of course, that has happened to most of us at some point. After about 30 seconds of non-response, you may as well just turn off the computer, because waiting 30 more years won’t get any response.

Although system lockups are relatively infrequent with Windows 2000, you have to handle the ones that you do experience. Because a Windows 2000 “lockup” might be the result of some heavily CPU-intensive task—not really a lockup at all—you need to wait a minute or two before deciding that your system is dead. If Windows 2000 still refuses to respond after a time, press Ctrl+Alt+Del and choose Task Manager. You can then choose an application and terminate that application, and your system will still work.

I still use some Windows 3.1 applications, mainly because no one’s converted them to Windows 9x or NT. One such application is cc:Mail, which now and then decides that whatever it’s doing is the most important thing that the operating system could ever imagine doing, and so cc:Mail puts the hourglass up on the screen. The hourglass, as you know, means “Wait while I finish what I’m doing.” You shouldn’t see too much of this behavior under a modern multitasking operating system. Anyway, cc:Mail uses the hourglass far too often, which caused me to look forward to Windows 95. I figured that the multitasking engine of Windows 95 would be able to tame cc:Mail’s penchant for the hourglass. Despite the advertising from Microsoft that Windows 95 would multitask Windows 3.1 programs better than Windows 3.1, I didn’t see any change. The cc:Mail program continued to display the hourglass just as it had under Windows 3.1—and Windows 95 ground to a halt.

I contacted someone inside Microsoft, asking why the hourglass was still around in cc:Mail when I ran it under Windows 95. “It’s just a temporary problem,” I was assured. “Temporary until when?” I asked. “Until you upgrade to a Windows 95 version of cc:Mail,” the Microsoft technician replied.

In contrast, I started up cc:Mail under Windows 2000 and told it to do something that would normally lead to an hourglass. The hourglass appeared, but only as long as I had the cursor over the cc:Mail window. When I moved the cursor to another program, it returned to its standard arrow shape, enabling me to continue working while cc:Mail computed.

In another case, I was sitting at an NT Server that acted as our World Wide Web server. Two people from outside my company were on our Web site, having accessed it from the Internet, and I was using our Internet connection to FTP to Novell, where I was downloading some drivers. The FTP transfer was going to take about 10 minutes, so I amused myself by starting up a Windows 3.1-based game. The game involves a lot of animation, and NT doesn’t run Windows 3.1-type animation very well. The game ran so slowly that I gave up and tried to exit from it.

The game didn’t want me to quit!

“Curses,” I muttered, and brought up Task Manager (remember, Ctrl+Alt+Del; then click Task Manager), clicked on the game, and clicked End Task.

NT thought about it for a minute and then said to me, “This program isn’t responding to my requests for it to terminate. Shall I just force it to shut down?” I thought, “Hey, I enjoy pushing around small, helpless applications as much as the next guy,” and so I told NT to go for it.

NT came back with more bad news. Basically, what it said was this: “I’ve tried to shut down the application, but it isn’t listening to me. What I can do as a kind of last resort is shut down the entire NT subsystem that runs Windows 3.1-type programs, a module called WOWEXEC. Should I?”

I hesitated. I had two outside customers on the Web server, which was a native NT application, and I was doing a lengthy FTP file transfer, which I could redo but didn’t want to. Also, WOWEXEC is a big subsystem. Would zapping it affect the Web or FTP sessions? I tried it.

In a matter of about 15 seconds (on a 25MHz computer), WOWEXEC was back up and running, the bad application was gone, and neither my Web nor FTP sessions were interrupted. Pretty neat.

TIP:  When you configure Windows 2000 to run Windows 3.1 programs, you can tell Windows 2000 to Run in Separate Memory Space. That command sets up a private copy of WOWEXEC for each old Windows 3.1 program. If you must pull the plug on a Windows 3.1 application, then you can do it without affecting any other Windows 3.1 or 32-bit application.
