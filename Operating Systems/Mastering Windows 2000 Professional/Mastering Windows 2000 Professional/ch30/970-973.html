Performance provides four views. You access these views by clicking the buttons on the left side of the toolbar. You can also click the View menu and select a view. The four views are simply different displays of the data that Performance can report, ranging from graphics to log files. Here’s a quick rundown on Performance’s views:

Chart  Displays counters graphically. You can add as many counters as you like, although more than about a half-dozen makes for a rather cluttered graph.
Histogram  The Histogram view is another form of the Chart view. Instead of a chart that display the historic data, the Histogram view displays bar graphs that dynamically update as the data changes.
Alert  Enables you to assign a threshold value (minimum or maximum) to a counter. You can specify that exceeding the threshold causes an event to be logged in Performance’s Alerts log or launches a program the first time the event occurs or every time the event occurs. The Alert view is accessed through the Performance Logs and Alerts branch of the console tree.
Log  Enables you to specify counters to write to a log file at intervals from 1 to 3600 seconds. The Log view can be accessed directly from Performance, or through the Performance Logs and Alerts branch of the console tree.
Report  Displays counters in real time (as Chart view does). Selected counters are listed and their values are updated at a specified rate.

Before getting into detail about each of these views, let’s look at some counters you’re most likely to monitor.

Windows 2000 classifies its subsystem components into objects, and each object has anywhere from two to more than a dozen counters that you can monitor. In addition, certain objects—such as processors, physical disks, and logical disks—also have instances because your Windows 2000 system may have more than one of these types of hardware. Suppose, for example, that you want to monitor part of a specific process (created when a program is running in Windows 2000). By selecting the process object, you’d see all processes currently running in the Instance field of the Add Counters view (shown in Figure 30.9). You could then select that process (in other words a specific instance of a process), and select which counter to monitor.


Figure 30.9  Each Windows 2000 object has a variety of counters, and some, such as CPUs, have instances because there may be more than one of that type of object.

Though more for network administrators, another cool feature is Performance’s capability to track counters on other systems on your network. For instance, if you found your machine reading and writing to your hard disk frequently or accessing the swap file, and you were curious to see if a coworker with a similar Windows 2000 system was experiencing the same situation, you could set a counter to watch that machine’s swap file activity and compare it with that of your own machine. To do so, you’d access the Profile System Performance on that person’s machine.

So without further ado, let’s take a look at several objects and some of their counters. Again, Performance can track hundreds of counters, and to detail every one of them would make this book an encyclopedia. Information about all counters is available within Performance if you really want to drill down and look at specific system components. You can do this by clicking the Explain button on the Add Counters dialog box.

Tables 30.1, 30.2, 30.3, and 30.4 show you the counters, their functions, and their importance for four objects: the paging file object, the processor object, the Logicaldisk object, and the server object.

Table 30.1 The Paging File (Swap File) Object
Counter Importance
% Usage A value that stays persistently high is one indicator that additional RAM would noticeably improve system performance, given the type of work you’re doing.

Table 30.2 The Processor Object
Counter Importance
% Processor Time May indicate that an application or a service is using the CPU excessively, cutting into overall system performance.
% Interrupt Time Interrupts execute in Windows 2000’s Privileged mode; applications execute in User mode. Excessive hardware interrupts can hamper overall system performance.
% Privileged Time Similar to % Interrupt Time; if your system is spending excessive amounts of time in Privileged mode, applications, which all run in User mode, have to wait for the CPU to return to User mode to access the CPU.
% User Time An indicator of how much CPU time applications have available to them.

Table 30.3 The Logicaldisk Object (Physicaldisk has Similar Counters)
Counter Importance
% Free Space Useful for setting an Alert threshold if this value dips below a certain value, say 20 percent.
Disk Read Bytes/sec Gives a general indication of a disk’s read performance.
Disk Write Bytes/sec Gives a general indication of a disk’s write performance.
Free Megabytes Similar to % Free Space, a different way of displaying the same information. Good for setting Alerts.

Table 30.4 The Server Object
Counter Importance
Bytes Total/sec Excessive network access of your system by other users can drag down your overall performance.
Error Logon Repeated failed logon attempts may mean a password-guessing program is trying to crack into your system.
Sessions Timed Out Useful for setting idle time-out values. Each active peer connection to your system slightly diminishes overall system performance.
