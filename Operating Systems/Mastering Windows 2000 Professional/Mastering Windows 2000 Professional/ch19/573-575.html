Reviewing Windows 2000 Security

If you’ve been a DOS, OS/2, Windows 9x, or Windows 3.x user, you’ll find that although you have to learn a fair number of concepts to be a Windows 2000 expert, most of them will be somewhat familiar because similar elements appear in all operating systems. For example, you have to learn about the Windows 2000 Registry, but Windows 95 has a Registry as well; you have to learn about attaching your workstation to your network, but the process is similar to attaching a DOS or Windows client to a network; and although the procedure for setting up new device drivers under Windows 2000 is unlike the procedure under any other operating system, it has some parallels to the way you do it in Windows for Workgroups. That’s all good news because it makes the job of becoming a Windows 2000 expert easier. But one concept about Windows 2000 is completely new and unfamiliar to the experienced PC user—security.

Unlike some other operating systems, Windows 2000 requires you to create a user account for yourself right on your PC before you can do anything on that PC. Yes, the idea that you must create your own user account on your personal PC before you can do anything with the PC is unusual—after all, most of us are accustomed to requiring network accounts, but not particular accounts on a workstation. But, as your father might say when you complain that something you don’t like isn’t fair—Get used to it!

The user account is an integral part of Windows 2000 and has some great benefits. For example, suppose you and Sue share a computer. You can set up the computer so that you own a directory on the hard disk and Sue owns another directory on the hard disk and it is completely impossible for Sue to access your data (and vice versa). Windows 2000 was built with security in mind, so exploit that power!

If you can understand how a lock and key provide security, then you can understand how Windows 2000 accomplishes security. Here’s the scenario: You’re sitting at your workstation running Word, you click File Open, and choose a file named report.doc. Word tries to read that file, and …

…Well, so far, this all sounds pretty normal; nothing is different here from what you have seen under Windows or DOS. But in Windows 2000, you can’t touch anything (a file, a printer, a directory) unless you have the proper permission to touch it. Every access is checked by a piece of Windows 2000 called the Security Reference Monitor (SRM). The SRM checks to see if you have the permissions to access report.doc and if you do, then Word will load report.doc.

The extremely compressed version of what’s going on under the hood is this: When you log on, you get a “key,” called a Security Access Token (SAT). Every object under Windows 2000 has a “lock,” called its Security Descriptor (SD). The key (SAT) must fit the lock (SD), or you don’t get to the file. This chapter shows you how to set up the keys and the locks so that you’re not constantly stumbling up against Windows 2000 security—while enabling you to protect your workstation’s data from unwanted access.

NOTE:  Many people use the term Access Token instead of the term Security Access Token (SAT) used in this book. They are one and the same.

Understanding User Accounts in Windows 2000

As you have just read, you must create separate user accounts on a Windows 2000 machine before any user can log on to the Windows 2000 workstation—and, unlike Windows 9x, Windows 2000 won’t let you get anywhere until you log on. You can’t avoid the logon screen. You create user accounts with a Control Panel applet called Users and Passwords, which you’ll meet later in this chapter. The user accounts sit in a database called the Security Accounts Manager, or SAM.

For simplicity’s sake, let us define some account terminology. The full name of a user account on an Windows 2000 Professional computer is the machine’s name, a backslash, and the username—for example, if your account name is Mark on a machine named M133, then your account’s full name is M133\Mark. Windows 2000 allows you to create user groups, which are accounts that contain other accounts. By default, every machine has a group named Users, and every user account is automatically a member of it (except the Guest account, which you’ll meet later in this chapter). Your M133\Mark account, then, is by default a member of M133\Users.

Accessing a File: Three Scenarios

Let’s consider now how Word reads a file. (Keep the lock-and-key analogy in mind.)

•  In the simplest case, you log on to your local PC and then try to access a file on your local computer.
•  In a somewhat more complex case, you might try to access a file on another computer via your network.
•  In the most complex case, you log on to a domain, rather than your local PC, and then access a file over the network.

Thankfully, these three scenarios have more similarities than differences, making them easy to understand.
