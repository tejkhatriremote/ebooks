Centralized Device Manager

The Windows 9x Plug-and-Play manager includes a tool called the Device Manager, and in my opinion it’s the single greatest operating system advance in Windows 95.

The Device Manager is so very important because it moves device drivers and hardware support out of the Dark Ages and into the modern age. In the Dark Ages, every program needed its own graphics drivers. A DOS program such as 1-2-3 or WordPerfect 5.x could exploit a fancy video board only if it had a special 1-2-3 video driver or a WordPerfect video driver. The same was true for printers; for example, you might use 1-2-3 and WordPerfect for DOS but have a printer driver only for WordPerfect—which creates a rather ridiculous situation wherein you could use your printer for one program but not for another.

The advent of advanced operating systems such as OS/2, Windows 3.1, and NT changed that situation because these operating systems all had shared display and printer drivers. Similarly, sound and network drivers meant that one driver offered sound and networking to all the operating system’s programs.

But the drivers weren’t centralized. No protocol kept the sound board’s driver from using a resource that the network board’s driver used. The reason for the potential chaos was that networking got attached to Windows 3.1 (or OS/2, for that matter) one piece at a time. Thus the drivers were never coordinated, making conflicts possible.

Windows 95 advanced the state of the art a bit by unifying all drivers, making them all register themselves with the Device Manager. For the first time in PC software, we actually have a program that can authoritatively provide a census of boards and the resources that the boards use.

Let me underscore the importance of this development. If you’ve ever conducted hardware configuration or memory management under DOS, then you know the pain of going to a strange machine and finding out what boards are in the PC and identifying the memory addresses, I/O addresses, IRQ levels, and DMA channels that the boards use. Sure, programs such as Microsoft’s MSD program or Quarterdeck’s Manifest program help, but all they really do is guess which resources a board uses; that’s right, they just guess. For example, one of the computers that I regularly use contains a video capture board, an Intel Smart Video Recorder Pro. It’s a nice board but it comes with Windows 3.1 drivers only; there are no DOS drivers. Because MSD and Manifest are DOS programs, they have no way to detect the board, much less a way to detect which IRQ the board uses. But the Device Manager puts an end to all that guesswork.

Device Manager is a great tool, and one that NT users will appreciate in Windows 2000. What’s more, the Device Manager in Windows 2000 is just one part of a comprehensive Hardware Wizard that will aid in installing, configuring, and troubleshooting all of your hardware devices.

Power Management

Running NT 4 on my laptop looked familiar, as I’ve used Windows 95 for some time. But something was missing, and it took me a little while to figure out that it was the little battery icon on the Taskbar.

NT 4 didn’t have any kind of Advanced Power Management (APM) support, but Windows 2000 does. This is one of the new features that will make it much easier to run this operating system on portable computers. In fact, I’m writing this section on my Dell Inspiron 7000, watching my battery meter very carefully.

Windows 2000 Weaknesses

Ah, but Windows 2000 is not all wine and roses. Windows 2000 isn’t right for everybody, at least not yet. Windows 2000 problems tend to fall into one of several categories:

Problems related to scale  Some problems stem from the fact that a lot fewer people are using NT than are using Windows or DOS. As a result, fewer people are writing applications and drivers for NT.
Problems related to expectations  Other problems come from people’s prior knowledge of Windows or Windows 9x. People expect Windows 2000 to have all the features of whatever operating system they already know, and Windows 2000 doesn’t always comply with that expectation.
Problems related to Windows 2000 security  Windows 2000’s middle name, security, gets it into trouble. Deciding to secure an operating system means that you can lock yourself out of your own computer. Believe it or not, it’s actually possible to lock Windows 2000 out of itself!
Problems related to Windows 2000’s legacy  It inherits some unpleasant characteristics from earlier operating systems.
Problems that aren’t really problems  Finally, some of Windows 2000’s “problems” are really shortcomings, things that it should be doing, but hasn’t gotten to yet.
