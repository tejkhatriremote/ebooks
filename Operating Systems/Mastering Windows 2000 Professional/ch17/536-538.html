No Bookmark Support Exists for Applications

Ever since Macintosh and Windows appeared, they—and subsequent graphical operating systems—have used the paradigm of a desktop. The basic idea has probably been to get across the notion that you get real work done with a modern operating system, hence the desktop metaphor.

The problem is, computer desktops don’t resemble my physical desktop at all. For one thing, when I leave for the day, my desktop pretty much stays the way it is. If I leave my stapler in the middle of the desktop, it doesn’t spirit itself off to the upper-left corner automatically.

For another thing, my “data files” and “applications” return each day to the place where I left them. Books open to page 45 are still at page 45 the next day; folders are open to the same sheet of paper. Half-finished drawings and my drawing tool (a pen) are waiting for my return.

Compare this real-life scenario to how Windows 2000 starts up in the morning. It remembers which folders you had open when you shut it down, and it re-opens them, but that’s it. You must restart all the applications by hand, tell them what files they were working on (or, of course, you can double-click the files themselves), and, after the files and apps are loaded, you still have to tell the application to go back to where you were yesterday.

Why not build a bookmark programming interface that application vendors could put into their applications, an interface that could remember where you left off when last running the application or, perhaps, the file? Then every data file could contain a bookmark pointing to where it was last edited. (After all, virtually every application keeps track of where you are while you’re running it; why not just save that information?) The Desktop would then only have to remember which data files you had open when you shut down. When the Desktop started up again, it would open the files, and their file associations would enable them to tell the Desktop which applications to start. Then the bookmark pointer would restore the application to exactly the state that existed when the computer powered down. One of these days, I guess …

Advice Before You Continue

By now you’ve read about the good news and the bad news in Windows 2000. On balance, Windows 2000 is the answer for many people and many applications, but not for all people. (You weren’t expecting me to try to tell you what to do, were you?) Remember that the choice of either the Windows 9x or Windows 2000 path can’t be a wrong one. If you haven’t yet committed yourself, this section offers a few words of advice.

Consult Microsoft’s Hardware Compatibility List (HCL)

Microsoft is in something of a tough spot when it comes to support. As a Microsoft VP once explained, “There is no such thing as PC-compatible,” meaning that every model has its own quirks. Because no two PCs are identical, building an operating system that will run on any PC is difficult for Microsoft (or anyone).

What Microsoft has done in response is to compile a list of machines that it has successfully tested and verified that Windows 2000 will run on. That list is called the Hardware Compatibility List, or HCL.

The HCL is a good idea but it has one problem: Getting on the list is expensive for a hardware vendor. A hardware vendor can’t just give a computer to Microsoft to be tested for Windows 2000 compatibility; Microsoft charges thousands of dollars to test a computer. The result is that you can find many perfectly good computers made by small firms that can’t afford to pay Microsoft to test their computers and so they never make it onto the HCL. If the computers don’t make it onto the HCL, then choosing one of them is basically a crap shoot. The only way you’ll know if Windows 2000 will run on the computer is to try and install it. At least with the HCL you can know ahead of time whether the computer will work.

Nevertheless, if you receive any kind of Microsoft product support for Windows 2000 (or if you intend to receive any kind of support), then you pretty much have to buy computers on the HCL. If you don’t, then Microsoft technical support people “get stupid” all of a sudden. You can ask a perfectly valid question of a Microsoft technical support person, something like, “What’s the difference between a workgroup and a domain in Windows 2000?” If the tech support person doesn’t know the answer, he’ll say, “Well, that depends. What kind of computer are you running this on?” If you answer, “I’ve got an Acme Thunderchip 586,” he’ll cluck his tongue and say, “Well, that’s not on the Hardware Compatibility List.” Ah, yes, the HCL—the last refuge of scoundrels.

NOTE:  If you work for a large company that has Microsoft Premier Support, then you’ve pretty much got to buy machines on the HCL. If you’re part of a smaller company, then you may choose to save some money by buying cheaper machines whose firms opted not to give Microsoft money to test their machines—but you’re removing yourself from a source of later support. Last time I checked, Microsoft Technical Support policy states that if your system isn’t on the HCL they can give you one hour and their best effort, or refund your money for the call and wish you luck.

By the way, you can find the most up-to-date copy of the HCL (I think Microsoft updates it at least twice a month) on Microsoft’s Web site, www.microsoft.com.
