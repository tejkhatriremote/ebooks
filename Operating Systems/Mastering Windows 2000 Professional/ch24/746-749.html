You’ve already seen that network shares have three types of permission levels: Read, Change, and Full Control. File and directory permissions offer more options; you can see those options in Table 24.3.

Table 24.3 File and Directory Permission Types
Type of permission Applied to a directory Applied to a file
Read Allows display of the file’s data, attributes, permissions, and owner.
Write Includes Read permissions, plus allows changes to the file’s data and attributes.
Execute Allows running of program files and display of attributes, permissions, and owner. Note it does not include Read permissions.
Delete Allows deletion of the directory. Allows deletion of the file.
Change Permissions Allows changes to the file’s permissions.
Take Ownership Allows changes to the file’s ownership.

Basically, the Change permission in share-level permissions is divided into three permissions under file and directory permissions:

Write  Lets you write files. As a side effect, it also lets you overwrite files, so by saving a new file on top of an old file you can delete it, even if you don’t have Delete permission.
Execute  Lets you execute a program, usually one with an .exe extension. If you do not have Execute permission, you can’t run a program even if you can see it.
Delete  Is intended to prevent you from deleting a file, but it seems useless; we’ve had no trouble deleting files that we did not have Delete permissions for. It does seem useful for directories, however; you can’t delete directories if you lack Delete permissions for that directory.

NOTE:  You don’t get much mileage out of giving someone Read and Write permissions and then not giving Execute permissions, because anyone who did want to run the program could just copy the file to a local directory. The copy would end up with the same permissions as everything else in the local directory (which probably includes Execute), and that person could run the copied file. What the Execute permission is useful for is the opposite application—give someone Execute access but no other access, and the person can run a program but not copy or modify it. The value is that people can run a program on the network, but not download it to their local machines—and perhaps to floppy disks or Zip drives—and from there steal the application.

NOTE:  If you assign someone Full Control permission for a folder, then give them No Access to a file in that folder, the user will be able to delete the file. This is the one loophole in the No Access permission and was designed into NTFS to support the requirements of the POSIX subsystem.

NOTE:  Permissions applied to a file take precedence over permissions applied to the folder. The assumption here is that if you care enough about the file to set permissions, they must be more important than the folder’s permissions.

File and Directory Permissions vs. Share Permissions

One important thing to understand about file and directory permissions is, as we’ve mentioned, that they work whether you’re networked or not. You access share-level permissions in the Sharing tab. Directory permissions show up in a separate tab labeled Security. To see the Security tab, right-click any file or directory on an NTFS volume from the My Computer or Explorer windows, and you’ll see a dialog box similar to the one in Figure 24.4.


Figure 24.4  Security tab of the Eudora Properties dialog box shows in this instance that Everyone has Full Control.

Even though Eudora is shared in the preceding example, it would have a Security tab whether it was shared or not. The Security tab displays the list of permitted users, as well as giving you the option of granting or denying access through each permission level as shown in Figure 24.4.

You may recall that the share-level permissions on Eudora was just Everyone/Full Control. But that Full Control is quite a bit different from the Full Control in the Directory Permissions dialog box—why? Understanding the answer to that question will drive home an important point about file and directory permissions.

The Eudora folder had file and directory permissions specified on it before it was shared. Merely sharing it and creating a list of share permissions does not affect the file and directory permissions at all. Here’s an extreme example: Suppose Eudora had file and directory permissions that allowed only a user named Meredith to access it. Then suppose Eudora was shared with share permissions that allowed only one other user, named Martin, to share access to Eudora. After Eudora was shared, no change would appear on the file and directory permissions for Eudora. Although the share-level permissions would allow Martin to access the file, the file and directory permissions would allow only Meredith in.

In this case, what would happen? Could Martin access Eudora? Could Meredith? Could both Martin and Meredith access Eudora or neither? The answer is, neither could access Eudora. When Meredith tried to attach to the share, the share-level permissions would keep her out. When Martin tried to attach to the share, the share-level permissions would give him the thumbs up, but the file system’s permissions would nix the connection.

Now, Meredith could physically walk over to the computer that Eudora sits on, log on, and access the directory. Because she wouldn’t tickle the network software, she wouldn’t be denied access.

To summarize the basic rule of data access on a network share that’s been formatted as NTFS: To access some data on the network, you must have share-level permissions as well as file and directory permissions for that data.
