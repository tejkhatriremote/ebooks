The Profile Folder Structure

The Profiles folder contains the ntuser.dat data that holds your environment settings preferences. In addition, a folder exists for each of the critical locations where data is stored. The folders are as follows (and shown in Figure 24.32):

•  Application Data
•  Cookies
•  Desktop
•  Favorites
•  Local Settings
•  My Documents
•  My Webs
•  NetHood (My Network Places)
•  PrintHood
•  Recent
•  SendTo
•  Start Menu
•  Templates
•  Windows


Figure 24.32  Standard User folder

NOTE:  The ntuser.dat and ntuser.dat.log files are the hives that create your HKEY_CURRENT_USER keys in the Registry. It’s not a good idea to try to change these files directly. Instead, use regedt32.exe to make changes. The ntuser.ini file keeps track of settings that do not need to be backed up during the normal logging procedures of the Registry.

Information that is stored in these files is used only for the individual. The power of this structure is the ability to copy information from one folder to another—to give other users certain features of your profile. In addition, it provides a straightforward method of backing up profile information to an external device or to a location on the network. If a user’s profile becomes corrupt, you can restore the part or parts of the profile that have been affected.

If you should happen to delete a Documents and Settings\userid folder, Windows 2000 will react as if the user has never logged on to the system before. It will create a new Profile folder for the user. This exercise can be very useful if the user has placed icons and information all over the Desktop and is now complaining that it keeps coming up as a mess. You can simply delete the folder in the Profiles directory, and a new one will be created the next time that that user logs on. It will be the default environment as created in the Default User folder.

Understanding Windows 2000 Profiles vs. 9x Profiles

If you’ve used Windows 9x, you know that it also has user profiles. Windows 9x stores a lot of its user environment information in a file called user.dat. A backup to this file, called user.da0, holds a copy of your Desktop, Start menu, etc. the last time that you successfully started your system.

If you are in an NT Server or NetWare environment, you can easily network your user profiles so that your wallpaper, icons, start menu, etc. can follow you around the network, regardless of which Windows 9x machine you sit at.

The user environment information for Windows 2000 Professional user profiles is stored in a file called ntuser.dat. You can also have roaming network profiles in Windows 2000 Professional as described for Windows 9x.

You cannot, however, mix and match. Windows 2000 profiles work only if you are sitting at a Windows 2000 machine, and Windows 9x profiles work only if you are sitting at a Windows 9x machine.

Receiving the Error Message “No Domain Server Was Available...”

When logging into the server, you may get an error message stating that no domain server is available to authorize you to log on to a domain. Often this error message will prevent you from seeing some resources in My Network Places and stop you from accessing the network entirely.

This message appears when the system is unable to contact a domain controller for the domain that you are logging on to. The obvious problem could be that the server is down. Another common cause of this error is that a bridge, router, or gateway is malfunctioning in your environment; so check the hub and the bridges to make sure that they are operating normally.

Easily the most common occurrence of this problem is when TCP/IP is the primary transport protocol in your environment. If the workstation is on one physical segment of the network and the domain controller is on another segment of the network, often the client machine is unable to see the server via NetBIOS. To test this situation, see if you can ping the server’s address. If you can ping the server but are unable to contact the server for domain logon authentication, you have determined the cause of your dilemma.

The most common solution is to install a Windows Internet Naming System Server (WINS Server) in your environment. It will handle the NetBIOS to IP address name resolution (in much the same way that a DNS server handles IP host name to IP address conversions). For more information on how to install the WINS Service on an NT Server, refer to Mastering Windows NT Server 4, Sixth Edition by Mark Minasi, published by Sybex.
