Full Screen or Window

You can make several choices on the Screen page. First, you need to decide whether to run the application as a full-screen or a window. Depending on the application, you may not have a choice; some applications, especially very graphically oriented ones using EGA or VGA, insist on their full screen. Some applications will let you set them to run in a window, but as soon as the program begins they revert to a full screen.

If you run the program in a window, you can set its size beforehand. If you preset the window size, you can’t resize it while the DOS session is running.

Window Appearance

In this section, you can use the Restore Settings on Startup parameter to determine whether all the appearance parameters that you’ve set—the font, window size, and window position—will be used the next time that you start the program.

Oddly enough, although these two settings apply only if you’re running the program in a window, they’re not grayed out even if Full Screen is selected.

Performance

You have two options for fine-tuning the performance of your application: Fast ROM Emulation and Dynamic Memory Allocation.

Fast ROM Emulation
Windows 2000 Professional contains some programs that perform the function of some of the BIOS routines found in your system ROM (such as display and keyboard controls). These programs often work better than the hardwired BIOS, but unless you tell DOS programs to use them, they’ll consult the instructions found in the BIOS. Enabling Fast ROM Emulation causes the program to write to the screen faster because the system will use the more efficient video-display routines in Windows 2000 rather than the ones in your system’s BIOS. For most applications, you won’t see a noticeable difference when you enable this option, but Windows 2000 does suggest that you leave it enabled unless your application uses nonstandard ROM functions or you’re having problems getting the application to write to the screen. For most applications, however, ROM emulation should present no problems.
Dynamic Memory Allocation
Some DOS applications have both text and graphics modes; one example is WordPerfect 5.1, which runs in text mode until you preview a print job and then it switches to graphics mode. Graphics mode requires more video memory than text mode. To allocate video memory to an application only as it’s needed, click Dynamic Memory Allocation, and Windows 2000 will supply the program with only as much video memory as it needs, leaving any extra for other programs. If you select this option, when an application switches from text mode to graphics mode, Windows 2000 will attempt to allocate more video memory to the application. If not enough video memory is available, then Windows 2000 won’t be able to run the program in graphics mode, and you may not be able to see your print preview, for example. Therefore, if you want to be sure that your program will work no matter what mode it’s in, make sure that this box is not checked.

NOTE:  Unchecking Dynamic Memory Allocation does the same thing as checking Retain Video Memory under Windows 3.x.

Foreground

If you check the Allow Screen Saver box (active by default), then the screen saver you chose in the Display section of the Control Panel will come up if you haven’t put anything into the system for a predetermined length of time.

Windowed and full-screen DOS applications recover from the screen saver somewhat differently. When you return to the screen by moving the mouse, a windowed application will be in the place you left it. If a full-screen application was active when the screen saver came up, however, then when you move the mouse to retire the screen saver, you’ll be back at Windows 2000’s Desktop with the application minimized in the toolbar. Click the application’s button on the toolbar to return to the application.

Background

If you click Always Suspend, Windows 2000 will cease the operation of the DOS application when it’s in the background. For example, if you’re calculating a spreadsheet in Excel and then fire up a DOS game like Warlords to while away the weary hours while the spreadsheet’s assembling, then the spreadsheet will stop while you’re playing Warlords.

This may seem like a bad thing, but an awful lot of programs don’t do anything when they’re in the background; they sit, waiting for some input from you. For example, if you’re working in a word processor, most of the time the system’s waiting for you, even if you’re a fast typist. (The only computer that we’ve ever seen that couldn’t keep up with a fairly speedy typist was an old computerized engraver that was something of a relic even in 1988, and comparing it to modern computers is like comparing a moped to the Concorde.) Even if a program isn’t doing anything except waiting, however, it’s still using up CPU cycles.

To get the most out of your CPU’s time, therefore, Windows 2000 can keep the program in a state of suspended animation until you express an interest in seeing it again. If you don’t want Windows 2000 to do this with one of your programs (such as that spreadsheet), then make sure that Always Suspend is not selected. This setting affects only the program whose properties you are adjusting.

Note that the Exclusive setting in the Windows 3.x PIF Editor is not part of Windows 2000. If you want a DOS program to get all the resources allocated to DOS programs when it’s in the foreground, you’ll have to check the Always Suspend option for every other program that is open at the same time.

NOTE:  If you’re accustomed to working with Windows 3.x’s PIF Editor, be careful with this setting. Under 3.x, selecting Background meant that the program would run in the background; under Windows 2000, it means that it won’t.
