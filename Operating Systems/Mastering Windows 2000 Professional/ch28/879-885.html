Auditing

Click the Advanced button to open the Access Control Settings dialog, then click the Auditing tab to display the Auditing Entries dialog box (shown in Figure 28.3), where you can keep track of who has accessed a file and when.


Figure 28.3  The Auditing tab enables you to track file usage.

When auditing a file, you first need to select the individual user or the group that you want to keep track of.

NOTE:  If you want to keep general information on who has accessed a file, select the group called Everyone and audit this group’s access to the file.
Turning Auditing On

In order for auditing to work, you need to turn on the auditing feature from the Computer Management console. Follow these steps:

1.  Open Administrative Tools in Control Panel, and double-click Local Security Policy.
2.  Expand the Local Policies entry in the console and select Audit Policy.
3.  Double-click the policy items you want to change to open their Properties dialog, then click either Success or Failure (or both) as the type of event to audit.

You can choose whether to audit the successes and/or failures. However, you cannot audit the successful file access unless you are also auditing the successful Object Access.

After selecting the users whom you want to audit, you specify what you want to audit—reads of the file, writes to the file, attempts to execute the file (if it is an .exe or .com file), and the like. You can audit either the successful or failed attempts to do the above, or both, because you can select these options separately.

Any events that you choose to audit will be placed in the Security log, which is part of the Event log. You can view these events from Administrative Tools Event Viewer.

Ownership

You can take ownership of a file as you can with any other resource in NT Workstation. To take ownership of a file, you must have the permission to do so. You would either have Full Control permission, or Special Access permissions which give you the permission to take ownership. When you take ownership, you will see the Owner dialog box shown in Figure 28.4.


Figure 28.4  The Owner tab lets you take ownership for a file or folder.

Remember, taking ownership of a file enables you to do anything you want with the file, no matter what your permissions are for that file.

Summary

The Summary tab is new in Windows 2000. This tab, shown in Figure 28.5, contains information on the file, such as a title, revision, and category.


Figure 28.5  The Summary tab contains information that distinguishes the file.

This information is more specific to documents, usually. A Word document, for example, would have a title and subject assigned to it, as well as a revision number, author, and more. This Summary of the document would server to fully distinguish the document from other documents in the same folder. Having the Summary tab for an executable file is less useful, but developers may begin to include information in their programs that will be displayed here.

Program

The Program tab confirms the paths to the file, as you can see in Figure 28.6.


Figure 28.6  Program tab of a Properties dialog box

At the top is the full name of the application. Because the full name doesn’t appear in the folder or at the top of the application window when it’s running, there’s little point in adjusting it. The next entries, however, are worth exploring:

Cmd Line
Indicates the name of the executable file for the program. If you change this entry, you won’t be able to run the program. Windows 2000 will automatically update this entry if you move a file, but be aware that not all programs will run from just any folder—homebodies, like Lotus 1-2-3, insist on being in their own program directories. A lot of DOS applications will work just fine if you provide the path in the form of a UNC. If you choose to go this route because the application is located somewhere on the network, please test the launching of the application; some DOS apps work fine with UNCs, and others do not like it at all.
Working
Indicates the directory where the application will go first to load and save data files. If you don’t enter anything here, the application will look first in its program directory. You can enter any drive or directory here, so if you store all your files in your home directory on the server, you can enter the directory information here.

TIP:  When specifying a working directory, make sure that directory will be available. If you specify a network directory (for example) and the network is not active, you’ll slow your application as it tries to connect to the unavailable directory.
Batch File
Enables you to specify a batch file to be run each time you start up the application. For example, the DOS disk copying program DCOPY won’t work unless you specify a source drive in the Startup command. Double-click the icon, and all you’ll get is a nastygram with instructions telling you how to start the program and the Help file. Therefore, to make this program work in Windows 2000, you must type DCOPY A in the Batch File box. From there, the program works as though you’d typed DCOPY A from the command line.

NOTE:  Make sure that you include the full path information if the batch file is not in the same directory as the program file.
Shortcut Key
Indicates a hotkey combination that enables you to start up an application without having to pull it from its folder or the toolbar. Shortcut keys must include Ctrl or Alt (for example, Ctrl+J) and cannot conflict with any other shortcut keys, including Windows 2000 combinations, or they won’t work. You cannot use the Tab, Esc, Enter, spacebar, Backspace, or Print Screen keys in a shortcut key.
Run
Specifies how the program should run: as a window, full screen (maximized), or minimized on the Desktop. Not all programs can run as a full screen, so the default Normal window option may be your best bet.
Close On Exit
Enables you to close the DOS windows when the application finishes running. This option is a good choice for most cases. However, at times you may be running a utility such as MEM, when you want to see the results of the MEM statement. The MEM statement takes only half a second to run, and when it is finished running it closes the DOS window behind it, so you don’t have enough time to view the result. To alleviate this problem, deselect Close on Exit, and the result will be put in an inactive DOS window—which means that the computer will take a snapshot of the final 25 lines of the MEM statement and hold it in a window for you to look at. When you are finished viewing the DOS window, simply close the window.

In addition to the main window, the Program tab is the starting point for a couple of advanced program settings: changing the program icon and setting the autoexec.bat and config.sys for this DOS session.
