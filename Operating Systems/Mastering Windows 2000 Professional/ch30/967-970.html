Upon opening Event Viewer, you’d see an Error event in the System log reported by Windows 2000’s Service Control Manager. After clicking it, you’d see the message “The following boot-start or system-start driver(s) failed to load: sndblst.” Although this somewhat terse message doesn’t provide any specifics, from Event Viewer, you could then check your Sound Blaster driver’s status in Control Panel. To do so, open the Control Panel, open the System applet and click the Hardware tab, then click the Device Manager button, and bring up the properties sheet for the SoundBlaster to check the settings.

Event Viewer has a few other handy features to check out.

From the Log menu, in addition to changing Log views, you can clear all events from all three logs. Be careful here, however. Although Windows 2000 will issue a warning, there’s no Undo for clearing the logs. If you think you want to access event information contained in the logs, save them as backups in a different subfolder so that you can access them later if need be.

You can tweak your log settings for all three logs, changing log size to be anywhere from 64KB on up (the default is 512KB) in the Maximum Log Size box in the Log Properties. You can also change the overwrite settings in the System Log Properties dialog box (see Figure 30.7). Windows 2000 defaults to overwriting events more than seven days old; so if your network administrator wants to back up your event logs weekly, all events will be catalogued. This can pose a problem only if your system is generating an unusually large amount of log events—for example, an application or device isn’t working properly, or a network protocol stack is continuously issuing warning events.

NOTE:  If you’ve set event overwriting to be cleared manually or have specified to overwrite events of a certain age, the Windows 2000 System log can fill up and can no longer be updated until you make more room in the log. If your system is having problems and can’t update the System log, diagnosing a problem may be more difficult. If your log file has filled, you can’t re-enable event logging by increasing the logs’ file sizes. You’ll have to back up your logs, clear all events from them, and then increase the files’ sizes.


Figure 30.7  Leave some kind of log overwriting enabled. If you’re concerned about being able to track events over a period of time, increase the file size of each log.

WARNING:  Be careful to watch the size of logs as they grow. If you have set the logs to never overwrite and they fill up, 2000 may hang. If you are auditing for security events, the recommended setting is to never overwrite because security events should be documented over long periods of time. Clearing and archiving the log should be a part of routine maintenance.

Using the View menu in Event Viewer, you can sort events chronologically, listing either the oldest or the newest events first. Also, you can find a specific event using Event Viewer’s search engine. This feature essentially duplicates the Filter Events function, except that Event Viewer’s search engine will display, one by one events, that fit a certain description rather than listing them all at once.

You can get details about a specific event by selecting Details from the View menu, although it’s probably easier to simply select the event and press Enter. Selecting Refresh updates the log you’re currently viewing. When you first launch Event Viewer, all three logs are updated. The log you’re viewing will not be updated, however, unless you select Refresh or view one of the other three logs. If you’re looking for a certain recent event and don’t see it, choose Refresh to see if it has been entered into the log you’re viewing.

Using the General tab of the Log Properties dialog box, you can enable the Low Speed Connection feature if you’re getting Event Log information about other computers (or they’re getting Event Log information about your computer) on your network and if you are connected to that network via a modem.

Running the Performance Management Console

Windows 2000 is designed to “self-tune” for optimal performance but also provides several tools for detecting possible performance bottlenecks. You can use the Windows 2000 Performance console to monitor and log the performance of hundreds of variables, including some very esoteric ones intended to be used by developers and network administrators when tracking application and system behavior. In addition to tracking performance on your system, you can track performance counters on other Windows 2000 machines on your network. We’ll take a look at the Performance console and some of its components. Rather than cover every variable counter you can assign in Performance, we’ll look at about a dozen or so key counters that deliver the most relevant system information. If you really want to dig deep into the more obscure performance variables that Performance can track, it gives you a brief description of what each function does.

NOTE:  If you want only a quick read on Windows 2000 “vitals,” right-click the Taskbar and select Task Manager. Though not new to Windows 2000, Task Manager has many added features, including a Performance tab that reports CPU usage and memory statistics. Task Manager also lists applications you have open, as well as processes that are running (those created by programs and Windows 2000 services). Task Manager is discussed in detail in the “Running Task Manager” section at the end of this chapter.

To launch Performance, click Start, select Programs Administrative Tools, and click the Performance icon. Windows 2000 displays the Performance window, as shown in Figure 30.8.
